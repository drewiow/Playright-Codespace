Advanced Mode Editor UX (Planned)

The runner UI may provide an expand / pop‑out control for the Advanced Steps editor.
The pop‑out view:Displays a larger textarea suitable for longer advanced workflows.
May include inline hints, examples, or templates for copy‑paste.
Shares the same underlying value as the inline Advanced Steps field.
Behaviour:The pop‑out is a pure UI affordance.
No change to submission format or execution semantics.
Opening or closing the pop‑out must not mutate step content.
Design intent:Reduce friction for longer advanced workflows.
Support discovery without constraining power users.
Keep the executor and backend completely unchanged.
Advanced Mode Authoring Model (Planned)
Advanced steps are authored using two complementary UI concepts, both of which resolve to plain advanced‑step text before execution.

Advanced TemplatesRepresent full, ready‑to‑run workflows.
Typically multi‑step and opinionated.
Selecting a template replaces the entire Advanced Steps editor contents.
Intended as starting points that users may optionally tweak.
Examples: Save and Exit, Save → Refresh → Exit.
Advanced ActionsRepresent atomic or conceptual helper operations.
May expand to multiple advanced steps internally (e.g. a complete logout sequence).
Actions are inserted at the cursor position (or appended) within the editor.
Intended to be composable building blocks for user‑authored workflows.
Examples: Save page, Logout, Open customer info.
Design rules:

Templates define a workflow; actions build workflows.
Both templates and actions are UI‑level affordances only.
The backend, parser, and executor remain unchanged and unaware of templates or actions.
All execution continues to operate on the final advanced‑step text supplied at run submission.

2026-04-20

Decision: Conditional env decryption
Details:
- decryptEnv is only invoked when ENV_ENC_PATH is present
- Providing ENV_ENC_PATH without ENV_PASSPHRASE is a hard error
- optionalEnvFile scripts must run without attempting decryption
Rationale:
- Allows true optional env support
- Prevents invalid crypto invocation
- Makes runner → script contract explicit

2026-04-20

Decision: Backend emits line-oriented logs
Details:
- stdout/stderr are buffered and split on newline before emission
- each emitted event represents one logical log line
- runner UI does not attempt to reconstruct lines
Rationale:
- stdout chunks are not line-safe
- ensures clean, readable real-time logs
- keeps UI rendering simple
2026-04-20

Decision: Advanced Actions are data-driven
Details:
- Advanced Actions are defined in a central registry
- UI dynamically renders action buttons from data
- No hard-coded HTML elements for individual actions
Rationale:
- Prevent UI duplication
- Simplify extension and maintenance
- Keep Advanced Mode declarative and scalable

2026‑04‑20
Decision: Advanced Steps token validation is UI‑only and CSV‑scoped
Details:

The Advanced Steps editor performs live, non‑blocking token validation.
Token syntax is {{TOKEN_NAME}}.
A token is considered supported if and only if its name matches a column header in the selected CSV.
Tokens not present in the CSV are visually highlighted in the editor.
Highlighting updates:

on every editor input (typing, paste, delete)
when CSV selection changes
when steps are inserted programmatically (actions, templates, token buttons)



Scope & Behaviour:

Highlighting is a pure UI affordance.
No validation errors are emitted to the backend.
No execution is blocked or altered by invalid tokens.
Backend, parser, and executor remain unaware of token highlighting.
Highlighting does not mutate editor content; it is rendered via a visual overlay only.

Rationale:

Provides immediate feedback for common authoring mistakes (typos, missing CSV columns).
Keeps execution semantics unchanged and predictable.
Reinforces the mental model that Advanced Steps resolve against CSV row data.
Aligns with the principle that the runner UI owns authoring UX while the backend remains deterministic.
